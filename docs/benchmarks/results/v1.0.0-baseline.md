# StreamForge v1.0.0 Benchmark Results

**Date:** 2026-04-18  
**Version:** 1.0.0  
**Hardware:** Local development machine  
**Sample Size:** 10 samples per benchmark  
**Rust:** 1.75+

## Filter Benchmarks

### Simple Filters (JSON Path Comparisons)

| Filter Type | Mean Time | Throughput |
|-------------|-----------|------------|
| Numeric comparison (`/field,>,value`) | 44.86 ns | ~22.3M ops/sec |
| String equality (`/field,==,value`) | 49.39 ns | ~20.2M ops/sec |
| Boolean check (`/field,==,true`) | 46.31 ns | ~21.6M ops/sec |

**Analysis:**
- Simple filters are extremely fast (~45-50ns)
- Single-threaded throughput: 20-22M operations/second
- JSON path resolution + comparison overhead minimal

### Boolean Logic Filters

| Filter Type | Mean Time | Throughput |
|-------------|-----------|------------|
| AND with 2 conditions | 105.51 ns | ~9.5M ops/sec |
| AND with 3 conditions | 147.49 ns | ~6.8M ops/sec |

**Analysis:**
- AND filters scale linearly with condition count
- 2 conditions ~2x slower than single (expected)
- 3 conditions ~3x slower than single (linear scaling)

### Regex Filters

| Filter Type | Mean Time | Throughput |
|-------------|-----------|------------|
| Simple regex (`^active`) | 55.85 ns | ~17.9M ops/sec |
| Email regex (complex pattern) | 48.61 ns | ~20.6M ops/sec |
| Version regex (`^2\.`) | 45.19 ns | ~22.1M ops/sec |

**Analysis:**
- Regex filters surprisingly fast (~45-56ns)
- Regex compilation is cached (one-time cost)
- Only matching time is measured
- Performance comparable to simple filters

## Transform Benchmarks

### Simple Transforms (Field Extraction)

| Transform Type | Mean Time | Throughput |
|----------------|-----------|------------|
| Extract single field (`/field`) | 916.73 ns | ~1.09M ops/sec |
| Extract object (`/object`) | 1.04 µs | ~960K ops/sec |
| Extract nested (`/data/nested/value`) | 916.73 ns | ~1.09M ops/sec |

**Analysis:**
- Extraction ~900ns-1µs (20x slower than filters)
- Involves JSON cloning (not zero-copy)
- Nested field extraction no slower than simple

### CONSTRUCT Transforms (Object Building)

| Transform Type | Mean Time | Throughput |
|----------------|-----------|------------|
| Small (2 fields) | 946.45 ns | ~1.06M ops/sec |
| Medium (4 fields) | 1.10 µs | ~909K ops/sec |
| Large (8 fields) | 1.43 µs | ~699K ops/sec |

**Analysis:**
- CONSTRUCT scales ~linearly with field count
- 2 fields: ~950ns
- 4 fields: ~1.1µs (2.3x fields = 1.16x time)
- 8 fields: ~1.4µs (4x fields = 1.5x time)
- Excellent scaling characteristics

## Parser Benchmarks

### DSL Parsing Performance

| Expression Type | Mean Time | Throughput |
|-----------------|-----------|------------|
| Simple filter (`/field,>,value`) | 100.18 ns | ~10M ops/sec |
| AND filter (`AND:/f1,>,v1:/f2,==,v2`) | 328.92 ns | ~3M ops/sec |
| Regex filter (complex pattern) | 409.52 µs | ~2.4K ops/sec |
| Array filter (`ARRAY_ALL:/arr,/f,>,v`) | 230.18 ns | ~4.3M ops/sec |

**Analysis:**
- Simple filter parsing: ~100ns (very fast)
- AND filter parsing: ~329ns (scales with conditions)
- **Regex parsing: 409µs (4000x slower!)** ⚠️
  - Regex compilation is expensive
  - Happens once per filter at startup
  - Not a runtime concern (compiled regex cached)
- Array filter: ~230ns (reasonable)

**Important:** Regex parsing overhead is one-time cost at config load. Runtime regex matching is ~50ns (see filter benchmarks above).

## Bottleneck Analysis

### Current Bottlenecks (in order):

1. **Transform cloning** (~900ns)
   - JSON cloning is expensive
   - Solution: Typed Envelope<K, V> in v1.1 (zero-copy)
   - Potential improvement: 3-4x faster

2. **Regex compilation** (409µs one-time)
   - Only affects startup time, not runtime
   - Low priority

3. **CONSTRUCT scaling** (linear with fields)
   - Expected behavior
   - Already efficient

4. **Filter evaluation** (~45-150ns)
   - Already very fast
   - Low priority for optimization

### Optimization Priorities (v1.1)

1. **High Priority:** Typed Envelope<K, V> (3-4x transform speedup)
2. **Medium Priority:** Zero-copy JSON paths (2x filter speedup)
3. **Low Priority:** Parser optimizations (startup time only)

## Recommendations

### For High Throughput (> 50K msg/s)

1. Use simple filters (avoid nested AND/OR if possible)
2. Minimize CONSTRUCT fields (< 4 fields)
3. Use passthrough when possible (no filter/transform)
4. Scale threads = CPU cores
5. Increase batch sizes

**Expected throughput:** 50-60K msg/s (8 threads)

### For Low Latency (< 10ms)

1. Use per-message commit
2. Small batch sizes (< 100)
3. Avoid complex transforms
4. 2 threads max (reduce contention)

**Expected latency:** 5-15ms p95

### For Balanced Performance

1. Use time-based commit (1 second)
2. Medium batches (2000 messages)
3. Simple to medium transforms (2-4 fields)
4. 4 threads

**Expected:** 20-30K msg/s, 50ms p95
